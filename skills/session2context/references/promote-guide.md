# 提炼操作规范（Promote Guide）

将 session 摘要中的候选项提炼到 `s2c-data/preferences/` 的操作规范。

> **核心原则：谨慎、透明、用户主导。**
> AI 只做提案和执行，判断权始终在用户手中。

---

## 目标文件：固定推荐集（唯一权威清单）

`preferences/` 下**推荐固定为以下 2 个文件**，提炼时只往里归类，**不随意造文件名**：

| 文件 | 本质 | 装什么 |
|------|------|--------|
| `profile.md` | 行为契约（用户） | 用户的稳定习惯与对 agent 的要求：沟通（语言/详略/格式）、协作流程（先计划后执行、改动前确认）、决策偏好、做/不做、个人一贯编码风格 |
| `proj-overview.md` | 项目知识（环境） | 当前项目/环境的事实：技术栈/包管理/框架、命令、目录结构、领域术语、本项目强制约定 |

归类规则：

1. 候选能归入 `profile` / `proj-overview` → 归入对应文件。
2. 归不进、但该条**确实重要且语义明显独立** → **提示用户是否新建**一个文件：
   ```
   这条候选不适合放进 profile.md 或 proj-overview.md：
   - <候选内容>
   建议新建文件 `<kebab-case>.md` 保存，是否创建？（是 / 否 / 改名）
   ```
   仅在用户确认后才新建；文件名用 kebab-case。
3. **不重要 / 临时性 → 不提炼**（直接丢弃）。

> 区分要点：`profile` = "agent 该怎么做"（行为）；`proj-overview` = "项目是什么"（事实）。编码风格若是用户一贯偏好放 `profile`，若是本项目强制规范放 `proj-overview`。

---

## 内容质量约束（每条都会进 prompt，务必精炼）

- **只收强信号**：用户明确纠正 / 反复强调 / 明确拒绝的才提炼；一次性、临时性的不收。
- **一条一义、一行写完**：祈使句、可直接执行，不写背景故事。
- **先合并后叠加**：新条目与现有条目语义相近时，在预览中提出**合并/改写**方案，不静默叠加。
- **总量上限**：每个文件 ≤ 5–8 条；超出时强制合并提炼，保持精炼。
- **价值自检**：写入前自问"这条值不值得每次进 prompt"，不值则不写。

---

## 文件内部排版（用 `##` 分类）

每个 preferences 文件内部用 `##` 小节**按类别分组**相关条目，不要把所有条目平铺成一长串。

推荐小节（可按需增减，不强制）：

- `profile.md`：`## 沟通`、`## 协作流程`、`## 偏好与禁忌`
- `proj-overview.md`：`## 技术栈`、`## 命令`、`## 结构与约定`、`## 领域知识`

好处：
- 文件本身可读、好维护，新增条目直接归入对应小节；
- Step 5 生成「文件描述」时可**直接按 `##` 小节名概括**，避免某个路径的描述过长、在 mdc / CLAUDE.md / AGENTS.md 里显得臃肿。

---

## 加载自检行（常驻，验证文件是否真的被加载）

每个**有真实条目**的偏好文件，末尾固定加一行加载自检（空桩 / 纯注释文件**不加**，以免被当成非空而注入）：

```markdown
> 加载自检：读到本文件，请在本次对话结束时输出一行 `✅ 已加载 <本文件名> —— <一句情话>`
```

- `<本文件名>` 用该文件自身名字（如 `profile.md`）。
- 作用：对话结尾出现哪几行 `✅ 已加载 …`，就说明本次实际读到了哪些文件；没出现即未加载，便于排查"用了哪些、漏了哪些"。
- 用 `>` 引用、**不用 `##`**，因此不影响 Step 5 按 `##` 小节生成的描述。

## 操作流程

### 1. 读取现有内容

写入前，先完整读取目标文件的当前内容。不能盲写。

### 2. 展示变更预览

在对话中展示将要做的修改，格式如下：

```
📄 准备修改：s2c-data/preferences/profile.md

当前内容（摘要）：
  [文件内现有的条目列表，若为空则显示"（空）"]

新增内容：
  + <用户的偏好描述 A>
  + <用户的偏好描述 B>

确认写入吗？（是 / 否 / 修改措辞后再写）
```

### 3. 等待用户明确确认

必须等到用户回复"是"或等效肯定词后才执行写入。

"好的"、"可以"、"写吧"、"确认" = 确认
"等等"、"改一下"、"不对" = 暂停，按用户要求调整

### 4. 写入规则

- 新条目追加到**对应 `##` 小节的末尾**；若无匹配小节，则在文件末尾新建一个合适的 `##` 小节后追加
- 加载自检行始终保持在**文件最后一行**，不被其他条目插到后面
- 去掉 preferences 文件中的注释占位符（`<!-- ... -->`），用实际内容替代
- 每个条目用 `- ` 开头，保持简洁（一行以内）
- **每次只写一个文件**，写完展示结果，再问下一个

### 5. 写入后确认

写入后，展示文件的**完整最新内容**，让用户核实结果：

```
✅ 已写入 s2c-data/preferences/profile.md

当前完整内容：
---
[文件全文]
---

是否继续处理下一个候选项？
```

---

## 边界与限制

- **不得自动写入**：任何对 `preferences/` 的修改都必须经过用户明确确认。
- **不得删除条目**：除非用户明确要求删除某条。
- **合并需确认**：新条目与现有条目语义相近时，可在变更预览中**提议合并/改写**，由用户决定；不得静默合并或静默叠加。
- **新建文件需确认**：默认只用 `profile.md` / `proj-overview.md`；仅当候选重要且归不进时，按上文「归类规则」提示用户确认后才新建（kebab-case）。
