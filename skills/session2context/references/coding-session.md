# 编码类 Session 分析维度

适用于：写代码、调试、重构、架构设计、代码审查等场景。

---

## 摘要模板

```markdown
# Session: <中文标题>（<日期>）

## 类型
编码类

## 本次任务
<一句话描述做了什么，例：为 rca_service.py 添加 VLLM 模型动态加载>

## 用户纠正 / 强调（最重要的信号）
<!-- 用户对 AI 输出的修正，是最强的习惯信号 -->
- <AI 的行为> → <用户要求改成什么>
- 示例：AI 在模块顶层初始化 DB → 用户要求改成 lazy 方式
（若无，写"无明显纠正"）

## 反复出现的偏好
<!-- 在 session 中多次提到或强调的要求 -->
- <偏好描述>
（若无，写"无"）

## 技术决策
<!-- 用户在多个方案中做出的选择 -->
- 选择：<方案/工具>（理由：<用户说的>）
- 拒绝：<方案/工具>（理由：<用户说的>）
（若无，写"无"）

## 工作流观察
<!-- 用户组织工作的方式 -->
- <观察到的工作习惯>
（若无，写"无"）

## 项目常识（新发现）
<!-- 本次 session 中出现的、值得记录的项目级知识 -->
<!-- 包括：常用命令、项目约定、目录约定、启动方式、特殊环境配置等 -->
<!-- 格式按内容复杂度选择：
     · 单条命令 / 简短约定 → 用 bullet：- <命令或约定>：<说明>
     · 多步操作 / 需要代码块的内容 → 用 ### 小节标题 + 自由格式 Markdown -->
- <命令或约定>：<说明>
（若无，写"无"）

## 待提炼到 preferences/ 的候选项
- [ ] <候选1>
- [ ] <候选2>
（仅列出候选信号，不指派目标文件；归类与文件由 Step 4 依 promote-guide.md 决定）
```

## 信号识别重点

### 强信号（几乎一定是真实偏好）
- 用户明确纠正 AI 的行为（"不对，应该..."）
- 用户在多个 session 中重复同样的要求
- 用户拒绝某个方案并给出理由

### 中等信号（可能是偏好，也可能是当前场景特殊）
- 用户选择了某个方案但未说明原因
- 用户要求某种代码风格但只提了一次
- 用户提到某个工具/库名称

### 弱信号（不要提炼）
- 用户询问某个概念（可能只是不了解，不是偏好）
- 一次性的特殊需求（"这次先这样"）
- 上下文相关的临时决策

> 本文件只服务 Step 1–2（判断类型 + 生成摘要）。候选项归入哪个 preferences 文件、是否新建，全部由 Step 4 依 `promote-guide.md` 决定，本文件不指派目标文件。
